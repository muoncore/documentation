The terms we use and what they mean

Muon Core

Muon is designed to let you build distributed systems that communicate using messages. In order to do that, you need a programming model inside your applications that understands the concepts you need and lets you do that easily.

Muon Core is a language/ runtime specific library that gives you those central concepts and bring them together into a coherent whole.

On top of Muon Core, you add Protocol Stacks to give yourself apis, Transports to move messages across a network and one or more Discovery systems to locate other systems.

Internally, it has a toolkit of components that let you construct messaging and transformation pipelines. Mostly, you won’t see these unless you build your own Stack, Transport or Discovery.

Protocol Stack ("a Stack")

A Protocol Stack, commonly known as "a stack" takes the low level apis from Muon core, that enable you to send messages to other systems and then converts those messages into ones that mean something.

You would build a Stack to implement a particular interaction between systems, and then add an API on top of that interaction.

The core pieces of a Stack are :-

  • The Client API. This is what your code will use to interact with the stack and trigger interactions. It should be language idiomatic and there will be a version of the API for every Muon supported language you also wish to support.

  • A Client Protocol. This piece manages muon messages. It takes data from the API, converts it into messages and instructs Muon Core to dispatch them to remote services. This protocol starts the conversation with a remote service. This can be coded in any language, with the option of using a JSProtocol to share the protocol code across runtimes

  • A Server Protocol. This, server side, protocol will interact with the client protocol. They must understand each others messages, be able to decode and manage the conversation between each other. The server protocol can then optionally access some local state to fulfill its part of the conversation. This can be coded in any language, with the option of using a JSProtocol to share the protocol code across runtimes

  • An optional Server API. It is likely that you want to provide developer facing API for the server side for them to interact and manage the state that the server protocol can see.

For example, in the RPC Stack :-

  • The client API is RpcClient.java and protocol.js

  • The client protocol is implemented as a JSProtocol, and is here

  • The server protocol is also a JSProtocol, and is here

  • The server API is a list of handler functions that are invoked by the server protocol, which you can see here and here

Protocol

A protocol is responsible for creating and receiving messages and interacting with an API as a appropriate. There is normally both a client and server side protocol for an interation. Each are referred to as a protocol, with the fully qualified name being "<protocol name> server protocol".

The phrase "RPC Protocol" refers to both the client and server of the RPC interaction code.

For example :-

  • The RPC client protocol, sends a message to the server, server receives and interprets it, sends one back.

  • The Reactive streams protocol converts the various RS signals into messages. The client and server each have responsibility for sending and understanding different sets of messages.

To implement a Protocol, you use the toolkit provided by Muon Core to obtain a channel to one or more remote systems, marshall messages and receive messages.

You can choose to implement the protocol in the same language as your runtime, or you can build it to be shared. A system has been developed that allows you to build protocols in javascript and have them run in any runtime that Muon supports. This is referred to as a JSProtocol. In this case, the same code is running to manage messages on all platforms. This will remove a common source of error between implementations, the incorrect marshalling of messages.

Discovery

A Discovery is a subsystem that enables Muon Core to locate which transport should be used to communicate with a given remote system.

Each Muon Core has a different API internally on what a Discovery is, however they all need to provide the same information, which is, given a query in the form of a logical service name or a set of tags to match, they should return one or more remote instances and the transport connection details that can be used to connect to those instances.

Transport

A transport has two jobs, to connect to remote services and to accept connections from remote services.

Once either of those has happened, then the transport creates a Channel abstraction to represent this connection, and then Muon Core (and above that, Stacks) then send messages to the remote service over these channels. Transports know very little about the messages they manage, only the routing headers on them.

This allows transports to be very differently implemented, since all they need to do is to create a bidirecitonal channel. The most common back end transport right now is AMQP, which creates two queues for each connection. Since it can manage the messages going over its connections, it actually fully multiplexes the channels over a single concrete link (think pipelining, but something that actually works).

Other implemented transports are websocket for Muon.js/ browser comms.

Future transports could be gRPC, Kafka, ZeroMQ, Aeron or anything that can send data in both directions at the same time in some manner between two processes.

Channel - "bi-channel"

A Channel is a core concept in the underlying Muon model that Muon Core creates. Muon is designed to make the interaction between systems obvious and portable (as opposed to, say, Actor systems that are designed to make internal state management obvious).

To do this, Muon Core creates a reduced form of Communication Sequential Processes (CSP), that allows you to form pipelines of code and connecting channels. The primary API that code talks to for message interaction then becomes the Channel Connection, and everything behind that connection is then masked from view.

This allows code to become very portable, testing is very easy and the system becomes a very modular beast.

This is implemented using Channel and ChannelConnection in Muon Java and channel.js (commonly referred to as bichannel) in Muon Node.

Codec

A common need when network transporting is to take an arbitrary object and convert it to binary for sending. This is done by a Codec implementation. Since Muon is aggressively cross platform, Codecs should also be cross platform. Right now we use JSON by default and Avro in some cases. The intention is to standardise on Avro as it becomes more widely implemented across the projects.

Other options that you could implement would be Protobuf, Thrift.

In addition to providing binary conversion, a Codec is also responsible for providing Schema/ type hint information for the system to use. For example, Avro can provide schema information for types and use that to perform mapping during decoding when there is a data version conflict.

The Muon introspection protocol can provide these schemas for remote systems to interrogate and use client side.
